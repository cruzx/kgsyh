# woodstuck Hook Design

这份文档定义 `woodstuck` 适合使用的钩子设计。

原则：

- 钩子只做 `收尾 / 提醒 / 分流 / 预检查`
- 复杂判断仍由 `woodstuck` 主体完成
- 能静默就静默，只有真正有价值时才打扰用户
- 默认优先做 `轻钩子`，不要把每个节点都自动化

## Hook 1: 任务分流钩子

### 目的

防止 `woodstuck` 每次都把整套上下文全读一遍。

### 触发时机

- 用户刚发起任务
- 命中 `woodstuck` 或明显进入其适用语境

### 输入

- 当前用户消息
- 当前上传稿件 / 图片 / 文件

### 动作

先判断主模式，只加载：

- `1 个主 playbook`
- 最多 `1 个 overlay`
- 必要时再补 `1 个 reference` 或 `1 个 research`

### 输出

- 内部路由结果
- 不一定对用户可见

### 建议实现

- 优先用 `SKILL.md` 内部规则实现
- 不需要额外 automation

### 成功标准

- 默认不再全读 `learning/*`
- 进入任务更快
- 回答更聚焦主矛盾

## Hook 2: 学习结果提醒钩子

### 目的

让用户能感知 `woodstuck` 真的在学习，而不是静默发生后完全无感。

### 触发时机

- 定时蒸馏完成后
- 或一次明确的学习沉淀动作完成后

### 输入

- 本轮新增的 `SESSION_LOG / PROFILE / TREND_LOG` 变更结果

### 动作

- 如果本次有新增耐久信息：
  - 用一句话通知用户 `这次新增了什么`
- 如果没有新增：
  - 保持静默

### 输出

- 一句简短提醒

### 建议实现

- 优先用 heartbeat automation

### 成功标准

- 用户能知道它学到了什么
- 但不会每次都被空提醒打断

## Hook 3: 出稿后复盘钩子

### 目的

把一次有效的收稿、改稿、判断，压缩成后续可复用的方法，而不是只完成当前回答。

### 触发时机

- 一次明显的出稿/改稿/收稿任务结束后
- 或用户明确确认某版方向成立后

### 输入

- 本轮主矛盾
- 用户接受/否定点
- 最终收法

### 动作

至少提炼：

- 这次真正解决了什么矛盾
- 哪个做法有效
- 哪个做法无效
- 是否值得升级为长期规则

### 输出

- `SESSION_LOG` 一条沉淀
- 必要时升级到 `PROFILE`

### 建议实现

- 优先由 `woodstuck` 在任务结束时自己触发
- 不建议做成每轮强制自动执行

### 成功标准

- 学到的是方法，不是流水账
- 不把一次性业务细节误记成长期规则

## Hook 4: Git 上传前检查钩子

### 目的

避免把脏文件、半成品、断链结构直接上传到仓库。

### 触发时机

- `18:30` 自动上传前
- 或用户手动要求上传前

### 输入

- 当前 git 改动
- 暂存区内容

### 动作

检查：

- 是否混入 `.DS_Store`
- 是否有明显无关文件
- `README / SKILL / playbooks / learning` 是否存在断链
- 是否有只改知识内容、没改索引说明的情况

### 输出

- 通过：继续上传
- 不通过：先提示原因，再决定是否拦截

### 建议实现

- 优先用 cron automation 内部检查
- 必要时配一个本地脚本

### 成功标准

- 推上去的是干净资产
- 不是一堆临时残留

## Hook 5: 资料包命中钩子

### 目的

让 `assets` 里的资料包真的参与判断，而不是只存放。

### 触发时机

- 用户提到 `概念版会员 / 会员特权 / 下载 / 音质 / 铃声 / 付费链路`

### 输入

- 当前任务关键词

### 动作

优先读：

- `assets/会员特权/README.md`

再按需分流到：

- `特权功能梳理.pdf`
- `概念版-会员特权表.xlsx`

### 输出

- 更准确的会员/特权判断

### 建议实现

- 直接写在 `SKILL.md` 路由规则里
- 不需要独立 automation

### 成功标准

- 资料包被真正调用
- 不再出现“明明有资料但没翻”的情况

## Hook 6: 联网学习筛选钩子

### 目的

只有在值得联网时才联网，避免每次都查外部资料。

### 触发时机

- 用户明确要最新 / 最近 / 现在
- 任务是趋势吸收、商业外扩、创业判断
- 任务明显依赖快变化领域

### 输入

- 当前任务类型

### 动作

- 命中则触发 `ONLINE_LEARNING_PROTOCOL`
- 未命中则只用本地知识

### 输出

- 是否联网学习的决策

### 建议实现

- 写在 `SKILL.md` 的加载规则
- 定时外部学习另由 automation 补充

### 成功标准

- 联网更准
- 不联网时也更快

## 推荐优先级

如果只做最有价值的一套，优先顺序是：

1. `任务分流钩子`
2. `学习结果提醒钩子`
3. `Git 上传前检查钩子`
4. `资料包命中钩子`
5. `出稿后复盘钩子`
6. `联网学习筛选钩子`

## 实施建议

### 适合直接写进 SKILL 的

- 任务分流钩子
- 资料包命中钩子
- 联网学习筛选钩子

### 适合用 automation 的

- 学习结果提醒钩子
- Git 上传前检查钩子

### 适合保留为人工触发 / 半自动的

- 出稿后复盘钩子

原因：

- 复盘质量依赖判断，不适合无脑全自动
- 强制每轮都跑，只会制造噪音
