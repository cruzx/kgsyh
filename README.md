# woodstuck

`woodstuck` 是一个面向设计判断、产品决策支持、商业化机会识别与方向推进的 Skill。它既能处理设计稿、交互稿、商业化页面和概念方案，也能继续往上看，帮助团队在复杂语境里完成判断、收敛分歧、识别机会，并推动下一步决策。

它不是单纯做“审美点评”，也不是只做“产品逻辑检查”。它更像一个结合了协作上下文、业务目标、用户研究、增长判断和组织经验的策略型推进工具。

## 这个 Skill 适合干嘛

它最适合这些场景：

1. 一个方向已经出现，但还没有形成足够清晰的判断
2. 需要把零散 idea、反馈和资料压缩成更可拍板的方案
3. 需要判断一个页面、入口、玩法或能力值不值得继续投入
4. 需要在用户体验、业务表达、落地成本和增长价值之间做平衡
5. 需要往当前页面外多看一层，找商业化、拉新、留存或功能外扩机会
6. 需要把复杂讨论收成更清楚的优先级、风险和下一步动作
7. 要结合历史评审语境、聊天反馈、参考基线和研究材料一起判断
8. 想让输出更像“能支持决策、能推动执行、能形成资产”的版本

## 它会优先帮你判断什么

这个 Skill 默认不会先讲一大段抽象道理，而是优先判断：

- 当前方案是不是顺场景、顺目标、顺组织语境
- 信息是不是更容易浏览、比较、判断和拍板
- 业务表达、用户价值和商业承接有没有说清楚
- 体验是不是耐看、可信、可持续，而不是只靠短时刺激
- 有没有更低风险、更快落地、更容易复用的推进路径
- 这是不是一个值得继续扩成模块、机制或方向的东西
- 现在最该补的是设计、交互、商业承接，还是更上层的产品判断

如果任务明显属于酷狗内部评审或推进语境，它会自动切到更偏 `决策收敛` 的判断方式：

- 先让方案 `成立`
- 再让方案 `有说服力`
- 最后让方案 `更容易被推进`

## 最常见的用法

### 1. 做设计判断

```text
帮我收一下这版，让它更容易过。
```

### 2. 做交互与链路判断

```text
帮我看看这版交互为什么还不成立，按更容易拍板的方式收一下。
```

### 3. 做商业化/增长判断

```text
这个会员入口值不值得做？帮我结合当前语境和已有触点一起判断。
```

### 4. 做概念方案收敛

```text
帮我把这版概念稿收成更稳、更耐看、也更像能上线的版本。
```

### 5. 做机会外扩判断

```text
别只看这一屏，帮我一起判断这件事有没有更大的产品和商业机会。
```

### 6. 吸收外部趋势再判断

```text
先联网看一下最近值得学的交互/视觉/产品案例，再帮我判断这版该往哪收。
```

### 7. 做方向或创业判断

```text
如果不只把它当一个页面，而是当一个产品机会来看，第一步该怎么做？
```

### 8. 基于参考语境做综合判断

```text
结合已有评审语境、点击基线和研究材料，帮我判断这件事下一步该怎么推进。
```

## 仓库结构

```text
.
├── SKILL.md
├── README.md
├── agents/
├── assets/
├── assets/竞品/
├── assets/会员特权/
├── benchmarks/
├── playbooks/
├── references/
├── learning/
└── research/
```

说明：

- `SKILL.md`：主 Skill，负责触发、路由、按需加载和输出纪律
- `references/`：评审路由、过稿镜片、轻量 spec 压缩规则
- `references/woodstuck-hook-design.md`：适合 `woodstuck` 的钩子设计方案
- `benchmarks/`：大盘数据、实验结果、改版复盘、点击基线、模块表现证据
- `playbooks/`：具体打法库，按任务类型拆开读取，如视觉收稿、交互收稿、商业外扩、img2、会员、概念版
- `learning/`：进化层，存稳定偏好、会话沉淀、趋势蒸馏、评分与迭代规则
- `research/`：按主题整理的压缩研究材料
- `assets/`：辅助理解语境的图、截图和资料包
- `assets/竞品/`：竞品截图归类资产区，按玩法结构而不是视觉风格存放页面参考
- `assets/会员特权/`：概念版会员特权资料包，包含 `特权功能梳理.pdf` 和 `概念版-会员特权表.xlsx`
- `agents/`：配套 agent 配置

## 它和普通评审 Skill 的区别

普通评审 Skill 往往只会说：

- 哪儿不好看
- 哪儿不统一
- 哪儿还可以优化

`woodstuck` 更关注的是：

- 为什么这件事现在还不够成立，或者还不够值得推进
- 卡点是在场景、信息、业务、视觉、链路，还是更上层的方向判断
- 现在最该改的是 `P0 / P1 / P2` 哪一层
- 有没有更稳的替代实现、更轻的验证路径或更强的组织说服力
- 这个点位/交互/页面还能不能长成更大的产品或商业机会
- 这件事现在该继续优化、补链路、做验证，还是直接上升到方向判断

也就是说，它不只是“帮助方案通过内部评审”的协作工具，更像一个在复杂语境里帮助团队做判断、推进决策、发现机会和沉淀方法的工作台。

## 自我进化机制

这份 Skill 现在带有一套显式的“双学习机制”，但主 Skill 已经收薄，不再默认每次把整套知识都读一遍：

- `learning/PROFILE.md`：长期稳定偏好
- `learning/EVOLUTION_RULES.md`：如何从一次对话提炼规则
- `learning/ONLINE_LEARNING_PROTOCOL.md`：什么时候该联网，联网后怎么蒸馏
- `learning/TREND_LOG.md`：外部最新信息里的高价值沉淀
- `learning/SESSION_LOG.md`：每次重要协作后的沉淀

默认加载策略现在是：

- 必读：`PROFILE.md` + `references/routing.md`
- 按需再读：`1 个 playbook + 1 个最相关 reference/research`
- 只有任务明显命中时，才补读 `SESSION_LOG`、`TREND_LOG`、点击基线或资料包

它的目标不是假装模型会自动永久学习，而是把两类东西都留下来：

- 你这边真实协作里反复成立的判断
- 外面最新产品、交互、科技、视觉趋势里真正值得学的部分

最后把它们压缩成更强的判断力、推进力、商业洞察力和方向决策力。

## 它现在不仅能做收敛，也能长判断

升级后，`woodstuck` 默认可以覆盖这些主模式：

1. `视觉收稿`
2. `交互收稿`
3. `商业外扩`
4. `趋势吸收`
5. `创业判断`

如果命中更具体任务，会优先转去对应 `playbook`，而不是把所有规则都压在主 Skill 里。

也就是说，你以后不只可以拿它优化一版方案，还可以拿它问：

- 现在外面有什么新机制值得学
- 这个需求有没有更大的商业价值
- 这个功能能不能扩成产品方向
- 如果离职创业，先做什么最值
- 现在该继续 polish 一页，还是该把问题提到更高层去重新判断
- 当前最值得推进的是方案优化、链路验证，还是方向下注

## 安装

把整个仓库放进你的 skill 目录，并保持目录名为 `woodstuck`：

```bash
git clone <your-repo-url> woodstuck
cp -R woodstuck ~/.codex/skills/woodstuck
```

如果你使用的是其他本地 skill 根目录，也可以保持同样结构复制进去。

如果你要把它安装到别人电脑，并希望对方也能继续“自主学习和进化”，再看：

- [DISTRIBUTION.md](DISTRIBUTION.md)

## Notes

- 这个 Skill 很依赖语境，所以它不是“通用美术点评器”，也不是“只会改稿的审稿器”，而是“带上下文的判断与推进器”。
- 如果任务不需要历史协作语境、内部评审风格或参考基线，这个 Skill 不一定是最合适的选择。
- 如果任务明确属于酷狗内部收稿语境，优先使用它会更稳。
