## 当前状态

### 业务判断要主动找规模和付费机会

- 场景：商业化设计、会员触点、概念版提案、增长相关交互或页面评审
- 稳定信号：用户不希望评审只盯某个单点入口或单个页面，而要求主动发掘更大的商业化机会和拉新机会
- 应对方式：默认同时判断 `当前点位成立度`、`可放大的拉新机会`、`可承接的付费机会`、`能否做成复用机制`；先看单点值不值得，再看能否扩成链路、模块或玩法
- 避免方式：不要只就一个按钮、一个弹窗、一个卡片讨论优劣；不要只谈转化，不谈规模；也不要只谈拉新，不谈后续付费承接
- 来源：2026-06-15 用户明确要求主动发掘商业化和拉新机会点，不要只盯单点
- 最近确认：2026-06-15

### 酷狗会员与概念版的目标区分

- 场景：会员需求、概念版需求、两者混合语境下的方案判断
- 稳定信号：用户明确区分两类目标：酷狗会员主要是引导用户付费；概念版除了扩大规模，还要增加用户付费
- 应对方式：评审前先判断当前任务属于 `会员付费型` 还是 `概念版规模+付费型`；会员需求优先看付费承接、权益表达、支付转化；概念版需求同时看规模扩张、分享传播、留存承接和付费增长
- 避免方式：不要把会员和概念版按同一套标准审；不要用纯会员思路去压概念版，也不要用只看规模的思路忽略概念版付费目标
- 来源：2026-06-15 用户明确说明“酷狗会员主要是引导用户付费，概念版要在扩大规模同时增加用户付费”
- 最近确认：2026-06-15

### 过稿要连细节一起过

- 场景：交互评审、商业化需求、会员链路、功能提案、过稿收口
- 稳定信号：用户不接受只判断大方向成立，而忽略细节分支、人群状态、下一步交互和承接闭环
- 应对方式：默认同时审 `方向成立度 + 人群分支 + 状态差异 + 下一步路径 + 成功失败承接`；尤其要单独检查非会员、已是会员但缺相关权益、已具备相关权益这类状态
- 避免方式：不要只看默认 happy path；不要只判断入口是否顺，不看落地页、CTA、成功/失败/返回路径；不要只说“方向对”却不补细节缺口
- 来源：2026-06-15 用户明确指出 mp3 下载需求中遗漏了非会员、已是会员但没下载券等情况，并要求过稿连同细节一起看
- 最近确认：2026-06-15

### 过稿要看商业外扩

- 场景：设计评审、交互收口、商业化页面、概念稿、功能提案
- 稳定信号：用户不只要当前页面或当前交互的成立度，还要求判断它有没有更强商业价值、能不能外扩到相关功能、能否形成更大的产品机制
- 应对方式：评审时默认拆成 `当前成立度 / 商业价值 / 外扩方向 / 扩展风险` 四层；不只看当前功能内是否顺，还要主动判断能否接分享、赠礼、会员感知、留存、设备联动、推荐或其他相邻能力
- 避免方式：不要只停在视觉和交互本身；不要把回答锁死在当前页；不要只说“这个点位行不行”，不说“它还能长成什么”
- 来源：2026-06-15 用户明确要求 kgsyh “不仅看当前设计，还要判断商业价值和相关功能扩展，聚焦当前功能外，看得更多更深”
- 最近确认：2026-06-15

### 收银台套餐主卡视觉收法

- 场景：会员收银台、生日套餐、节日礼包、商业化套餐主卡的视觉收稿
- 稳定信号：用户更接受“一体化、像商品的套餐主卡”，不接受上半张氛围海报加下半张价格卡的拼接感；允许稍微热闹，但不能兴奋过头
- 应对方式：先把主卡收成一个完整容器，再按 `限定角标 -> 标题 -> 主视觉 -> 价格利益点 -> CTA -> 套餐内容` 建立连续视觉流；热闹感优先加在局部彩屑、柔光、庆生碎点上，再做一轮去兴奋化压回正式可上线稿
- 避免方式：不要把蛋糕、价格、按钮切成两段；不要靠过强发光、礼盒装饰、彩屑堆热闹；不要把页面做成生日活动海报而弱化会员套餐商品感
- 来源：2026-06-15 用户连续收生日会员套餐收银台视觉，明确要求“做成整体”“再热闹一点点”，并继续否定过度兴奋化版本
- 最近确认：2026-06-15

### 会员特权点位收法

- 场景：会员触点、商业化页面、概念版里的特权引导点位判断与收稿
- 稳定信号：用户更接受“特权像当前内容或当前能力的自然延伸”，不接受额外硬塞一个独立入口、横幅或泛会员条
- 应对方式：优先判断当前页面的主场景，再把特权收成 `内容结果`、`能力升级` 或 `当前动作的顺手下一步`；首页偏轻引导，详情页/播放页/功能面板承接主转化
- 避免方式：不要只做“哪里还能再加一个点位”；不要写泛泛的“更多特权/会员中心”；不要让会员条跳出当前容器、打断浏览或播放叙事
- 来源：2026-06-15 用户连续评审多个会员特权点位页面，明确否定生硬加入口，更认可内容化特权、音质能力升级、预约/收听链路承接
- 最近确认：2026-06-15

### 更高完稿要求

- 场景：重要稿件、概念版、需要交稿或重画的任务
- 稳定信号：用户希望结果不只是“能用”，而是尽量接近非常完美，并且更有创造力
- 应对方式：默认增加自我评分、创意分叉、交付前反驳检查，必要时同时准备“更稳版”和“更出彩版”
- 避免方式：不要只给保守收敛答案；不要把“创意”误做成堆砌；不要停在第一版就结束
- 来源：2026-06-14 用户明确要求 skill 进化到每次稿子都更完美、更有创造力
- 最近确认：2026-06-14
